# Company Report Plugin

ChatGPT Work와 Codex에서 회사 보고용 16:9 슬라이드를 만드는 스킬 전용 플러그인입니다. 요구사항 정리, 구조 승인, PNG 시안 승인, 편집 가능한 PPTX 제공을 순서대로 수행합니다.

보고서를 단순히 예쁘게 배치하는 데 그치지 않고, `문서명 → 핵심 결론·판단 → 근거·세부 → 다음 조치` 순서로 읽히는 1페이지 보고서를 만드는 것이 목표입니다.

## 지원 환경

| 환경 | 지원 | 비고 |
| --- | --- | --- |
| ChatGPT 웹 Work 모드 | 지원 | 설치 또는 공유된 플러그인 사용 |
| ChatGPT 데스크톱 Work/Codex | 지원 | 로컬·GitHub 마켓플레이스 설치 가능 |
| Codex CLI | 지원 | `/plugins`에서 설치 |
| 일반 Chat 모드·모바일 | 미지원 | 현재 플러그인 실행 표면이 아님 |

플러그인은 로컬 서버, MCP 서버, 외부 API를 요구하지 않습니다. 레퍼런스, 범용 PowerPoint 템플릿, 나눔바른고딕 OTF Regular/Bold는 플러그인에 포함되며, 사용자 회사 템플릿과 근거 자료는 대화 첨부 파일로 받습니다.

## 제작 흐름

두 번의 명시적 승인 게이트를 포함한 다음 순서를 지킵니다.

1. **요구사항 정리**: 보고 대상, 핵심 결론, 판단 질문, 근거, 분량, 디자인 기준을 짧은 브리프로 정리합니다.
2. **구조 제안·승인**: 보고 유형과 조합 레시피, 최상위 영역과 하위 항목을 텍스트로 제안합니다. 이 단계에서 승인받기 전에는 문구와 슬라이드 원본을 만들지 않습니다.
3. **편집 가능한 원본 작성**: 승인된 구조에 실제 문구를 채우고 텍스트, 도형, 표, 차트를 가능한 한 네이티브 객체로 유지합니다.
4. **PNG 렌더링·검수**: 첫 렌더링은 바로 공개하지 않고 1280×720 전체 화면에서 메시지, 줄바꿈, 겹침, 여백, 강조를 검사하고 수정합니다.
5. **PNG 시안 승인**: 검수를 통과한 PNG만 제시합니다. 수정 요청이 있으면 같은 편집 원본을 고쳐 다시 승인받습니다.
6. **PPTX 내보내기·재검수**: 승인된 PNG를 만든 동일한 원본에서 PPTX를 내보내고 다시 렌더링해 구도와 줄바꿈이 승인본과 일치하는지 확인합니다.

`승인`, `확정`, `이대로 PPTX 만들어줘`처럼 분명한 표현만 승인으로 간주합니다. 질문이나 일부 수정 의견은 승인으로 처리하지 않습니다.

## 지원 보고 유형과 레시피

| 보고 유형 | 기본 레시피 | 용도 |
| --- | --- | --- |
| 신규 기획·제안 | `R04` | 적용 범위, 추진 구조, 목표와 승인 요청 제시 |
| 정기 진행 보고 | `R01` 변형 | 계획 대비 상태, 진행 경과, 이슈와 다음 우선순위 공유 |
| 문제·이슈 보고 | `R01`, `R01-G2` | 한계와 영향을 설명하고 지속·보완·전환 판단 지원 |
| 결과·성과 보고 | `R01`, `R02` | 수행 결과, 한계와 후속 방향 합의 |
| 업무 개선 보고 | `R03-S`, `R03-M` | 단일 또는 복수 과제의 현재 방식, 변경 사항, 효과와 수행 구조 설명 |
| 의사결정 요청 | `R05` | 선택지, 판단 기준, 권고안과 결정 후 행동 제시 |
| 백데이터 | `R06` | 요약 장표의 결론을 검증하는 상세 표·차트 제공 |

진행 타임라인과 상세 이슈 두 개가 하나의 공통 판단을 뒷받침하면 `R01-G2`를 사용합니다. 이때 `1:2`는 왼쪽 타임라인과 오른쪽 이슈 그룹이라는 **두 개의 최상위 영역**을 뜻합니다. 오른쪽의 두 이슈는 `G01` 그룹 안에서 내부 구분선과 번호형 소제목으로 나누며, 세 개의 동급 카드처럼 만들지 않습니다.

## 주요 동작

- 기본 결과물은 1페이지 요약 1장입니다.
- 필요한 경우에만 백데이터 슬라이드를 제안합니다.
- 제목 아래 첫 본문에는 현재 상태와 의미, 다음 방향을 담은 핵심 결론·판단을 먼저 둡니다. 화면 라벨에 `보고 목적` 또는 `[보고 목적]`을 사용하지 않습니다.
- 구조를 먼저 텍스트로 제시하고 명시적 승인을 받은 뒤 문구와 슬라이드 원본을 작성합니다.
- 같은 편집 가능한 원본에서 PNG와 PPTX를 생성합니다.
- PNG 승인 전에는 최종 PPTX를 제공하지 않습니다.
- 웹에서는 PNG와 PPTX를 대화에서 열 수 있는 첨부 파일로 제공합니다.
- 웹 작업 세션에 번들 글꼴을 등록하고 실제 글꼴 이름을 확인한 뒤 렌더링합니다.
- 번들 템플릿은 표지 없이 6개의 범용 조합 레시피 장표로 구성하며, 단일 과제 장표는 개선 요약·일정과 실제 수행 단계·입력·처리·산출을 한 화면에 함께 설명합니다. 실제 회사 로고, 고객 정보, 사내 화면은 포함하지 않습니다.
- 각 템플릿 장표의 객체 이름과 ID를 분리해 웹 미리보기에서 과제 밴드, 화살표, 노드가 다른 장에 섞이지 않도록 구성합니다.
- 보고 상황에 맞는 의미 기반 컴포넌트 조합 레시피를 선택하고, 번들 템플릿을 완성 형태의 시각 기준으로 사용합니다.
- 처음 보는 청중도 한 장에서 배경·수행·결과·영향·다음 조치를 이해하도록 구체적인 문장과 근거를 채우며, 검증된 정보가 있는데 큰 빈 공간을 남기지 않습니다.
- 적용·완료·개선 조치는 조치명만 나열하지 않고 확인된 개선 결과와 연결합니다. 아직 검증되지 않았으면 기대효과와 재검증 계획을 함께 표시합니다.
- 원문에 구체 사례가 있으면 관련 문장, 사례 영역 또는 백데이터에 포함하며 근거에 없는 사례나 성과를 만들지 않습니다.
- 날짜는 정보 정밀도에 따라 `'26.05月`, `2026-05-21`, `03/26` 형식을 사용하고 같은 타임라인·표·목록에서는 형식을 통일합니다. `초·중순·말`처럼 근사 시점만 제공된 경우 임의의 일자를 만들지 않습니다.
- 어려운 약어·전문용어는 대상 바로 옆에 초록색 `※ [용어]: [쉬운 정의]`로 설명합니다.
- 생성 시 핵심 구문·밑줄·영역 테두리는 빨강, 첨부 참고자료는 초록 `※`, 부가 설명은 작은 회색 문구로 구분합니다. 필요한 도식·flowchart·화면 캡처 자료가 없으면 편집 가능한 사각형 자리만 확보하며, 해당 요소들은 템플릿에 상시 자리표시자로 두지 않습니다.

## 품질 검수

사용자에게 PNG를 제시하기 전과 최종 PPTX를 내보낸 후에 다음 항목을 모두 확인합니다.

1. **메시지·컴포넌트**: 제목 다음에 핵심 판단이 먼저 나오고, 선택한 레시피와 정보 위계가 유지되는지 확인합니다.
2. **줄바꿈·표기**: 제목, 탭, 날짜, 전문용어, 결론이 의도치 않게 분리되지 않았는지 확인합니다.
3. **겹침·경계**: 텍스트, 도형, 탭, 그림자, 구분선, 타임라인 노드의 겹침과 슬라이드 밖 넘침을 확인합니다.
4. **여백·정렬**: 좌우 안전 여백, 열 간격, 패널 패딩, 기준선과 빈 공간의 균형을 확인합니다.
5. **강조·색상**: 빨간 밑줄의 위치와 길이, 네이비·빨강·초록·회색의 의미가 일관적인지 확인합니다.

Vision 검수와 별도로 글꼴 등록·조회, 객체 경계, 미해결 자리표시자, 편집 가능 여부를 검사합니다.

## 사용 예시

플러그인을 선택한 뒤 보고 목적과 알고 있는 사실을 자연어로 전달하면 됩니다. 보고 대상이나 디자인 기준이 없으면 결과를 크게 바꾸지 않는 범위에서 합리적으로 가정하고 진행합니다.

```text
QA 과제 완료 보고를 1장으로 만들어줘.
개발은 완료했지만 LLM과 OCR 한계로 현업 품질 기준을 충족하지 못했어.
왼쪽에는 추진 경과, 오른쪽에는 미해결 이슈 두 개, 하단에는 KT AICC 과제에서
재설계·재검증한다는 결론을 넣어줘.
```

일반적인 진행 보고도 다음처럼 요청할 수 있습니다.

```text
경영진용 1페이지 진행 보고서를 만들어줘.
계획 대비 상태, 이번 달 주요 경과, 핵심 이슈와 다음 달 우선순위를 넣어줘.
```

스킬은 먼저 구조안을 제시하고 승인을 요청합니다. 구조 승인 후에는 검수를 통과한 PNG 시안을 보여주며, PNG까지 승인받은 뒤에만 최종 PPTX를 제공합니다.

## 글꼴 동작

웹 호스트에서는 스킬의 `assets/fonts`에 포함된 나눔바른고딕을 현재 작업 세션의 사용자 글꼴 영역에 등록합니다. 영구 시스템 설치는 요구하지 않습니다. 등록이나 확인 기능이 없는 호스트에서는 임의 대체하지 않고 사용자에게 제한을 알립니다.

PPTX 내보내기가 글꼴 파일 자체의 내장을 보장하지 않으므로, 파일을 받은 PC에서도 나눔바른고딕을 설치해야 승인된 PNG와 같은 모양을 유지할 수 있습니다. 글꼴은 SIL Open Font License 1.1에 따라 재배포하며 전문과 고지문을 `assets/fonts/OFL.txt`에 포함합니다.

## ChatGPT 웹에서 사용

ChatGPT 웹 사용자는 Work 모드의 **Plugins**에서 설치된 플러그인을 사용합니다. 로컬 파일 경로를 웹 브라우저에 연결할 필요는 없습니다.

### 워크스페이스 내부 배포

관리자 또는 제작자가 한 번만 다음 절차를 수행합니다.

1. 이 저장소를 GitHub에 게시합니다.
2. ChatGPT 데스크톱 또는 Codex CLI에서 GitHub 마켓플레이스를 등록합니다.

```bash
codex plugin marketplace add OWNER/REPOSITORY --ref main
```

3. ChatGPT 데스크톱의 Work 또는 Codex에서 **Plugins**를 열어 `Company Report`를 설치합니다.
4. 플러그인 상세 화면에서 워크스페이스 구성원이나 그룹에 공유합니다.

웹 사용자는 다음 단계만 수행합니다.

1. ChatGPT 웹에서 Work 모드로 전환합니다.
2. **Plugins → Shared with me**에서 `Company Report`를 설치합니다.
3. 새 Work 대화를 열고 `@`로 플러그인을 선택합니다.

```text
경영진용 1페이지 진행 보고서를 만들어줘.
```

### 공개 Plugins Directory 배포

불특정 ChatGPT 웹 사용자가 검색·설치하게 하려면 OpenAI Platform의 플러그인 제출 포털에서 **Skills only** 유형으로 제출합니다.

- 제출 포털: <https://platform.openai.com/plugins>
- 공식 안내: <https://learn.chatgpt.com/docs/submit-plugins>

공개 제출에는 검증된 개발자·사업자 정보, 로고, 웹사이트, 지원 URL, 개인정보처리방침, 이용약관, 시작 프롬프트, 테스트 케이스가 추가로 필요합니다. 이 저장소에는 임의의 신원이나 URL을 넣지 않았습니다.

## 로컬 설치와 테스트

GitHub 게시 전에는 저장소 루트를 로컬 마켓플레이스로 등록할 수 있습니다.

```bash
codex plugin marketplace add .
codex plugin add company-report@company-report-marketplace
```

설치 또는 업데이트 후에는 반드시 새 대화나 새 CLI 세션에서 테스트합니다.

## 공개 전 확인

- `plugin.json`의 작성자 이름을 실제 배포 주체에 맞게 수정합니다.
- 공개 제출용 웹사이트, 지원, 개인정보처리방침, 이용약관 URL을 준비합니다.
- 템플릿과 문서에 회사명, 로고, 고객명, 이메일, 전화번호, 주소가 없는지 확인합니다.
- 번들 글꼴이 공식 배포본인지, `OFL.txt`와 함께 배포되는지 확인합니다.
- 커밋 기록과 Git 설정에 토큰, 계정명, 내부 경로가 없는지 확인합니다.
