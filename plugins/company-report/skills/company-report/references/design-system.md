# Design System

실제 보고서 스크린샷에서 추출한 시각 문법을 공개 가능한 범용 규칙으로 정리한다. 원본 화면, 회사명, 로고, 프로젝트명, 고객 정보는 저장하지 않는다.

## 목차

1. 우선순위와 캔버스
2. 제목과 공통 크롬
3. 색상과 타이포그래피
4. 정보 밀도와 설명 수준
5. 날짜 표기
6. 항목, 박스, 표
7. 강조, 참고, 예시, 시각자료
8. 컴포넌트 조합 시각 규칙
9. 검수 기준

## 우선순위와 캔버스

1. 사용자가 제공한 실제 회사 PPTX와 브랜드 가이드
2. 사용자가 승인한 별도 디자인 지시
3. 번들 `assets/template.pptx`와 이 문서

- 화면 비율: 16:9, 기본 캔버스 1280 × 720px
- 좌우 안전 여백: 56~64px
- 상단 제목 기준선: 36~48px
- 빨간 구분선: y 104~116px, 높이 6~8px
- 본문 시작선: 142~160px
- 하단 안전 여백: 40~48px
- 한 슬라이드의 핵심 결론·판단은 하나로 제한한다.
- 실제 보고서는 정보 밀도가 높아도 되지만 제목, 요약, 근거, 조치의 위계는 즉시 보여야 한다.

## 제목과 공통 크롬

- 상단 제목은 업무명·과제명·계획명처럼 문서를 식별하는 짧은 명사형 문서명으로 쓴다.
- 제목은 왼쪽 안전 여백에서 바로 시작하고 불릿, 사각형, 번호 표식을 붙이지 않는다.
- 제목 아래에는 슬라이드 폭 대부분을 차지하는 굵은 빨간 가로선을 둔다.
- 왼쪽 가장자리의 짧은 빨간 세로선은 유지하되 제목 사각형과 경쟁하지 않게 한다.
- 제목을 결론, 주장, 원인·결과를 모두 담은 긴 서술문으로 만들지 않는다.
- 빨간 구분선 아래 첫 본문에는 핵심 결론 또는 판단을 1~2줄로 먼저 배치하고, 그 아래에 근거·세부·다음 조치를 둔다.
- 화면 라벨에 `보고 목적` 또는 `[보고 목적]`을 쓰지 않는다. 라벨이 필요하면 `핵심 결론` 또는 `핵심 판단`을 사용한다.
- 기본 1페이지 보고서에는 별도 표지를 만들지 않는다. 여러 장일 때만 표지를 선택적으로 사용한다.
- 실제 로고가 없으면 임의 로고를 만들지 않고 텍스트 푸터와 페이지 번호만 둔다.

## 색상

- 배경: `#FFFFFF`
- 기본 텍스트: `#111111`
- 보조 텍스트: `#606368`
- 네이비 구조색: `#052B68`
- 빨간 제목·공통 크롬: `#E00012`
- 빨간 내용 강조·밑줄·영역 테두리: `#C00000`
- 파란 과제·단계 강조: `#001EFF`
- 녹색 참고자료·용어 주석: `#3B7D23`
- 회색 부가 설명: `#A6A6A6`
- 회색 패널: `#F1F1F1`
- 구분선: `#C9CBCD`
- 노란 표식: `#FFF200`, 전환점이나 `TO-BE`처럼 매우 제한적으로 사용한다.
- 네이비는 구조 라벨, `#E00012`는 공통 크롬, `#C00000`은 핵심 내용 강조, 파랑은 과제명, 초록은 참고자료·용어 정의, 회색은 부가 설명에만 사용한다.
- 기존 보라색 하단 장식선은 사용하지 않는다.

## 타이포그래피

- 선호 서체: 나눔바른고딕 OTF (`NanumBarunGothicOTF`)
- 번들 굵기: Regular, Bold
- 표지 제목: Bold 34pt
- 본문 슬라이드 제목: Bold 30pt
- 핵심 메시지: Regular/Bold 18~20pt
- 네이비 탭·큰 구조 라벨: Bold 15~18pt
- 항목 제목: Bold 15~17pt
- 1단계 본문: Regular 11~12pt
- 2단계 본문·표: Regular 9.5~10pt
- 주석·출처·페이지: Regular 8.5~9pt
- 좁은 타임라인의 날짜(`'26.05月`, `YYYY-MM-DD`, `MM/DD`): Bold 8pt, 한 줄 전용 영역 사용
- 핵심 본문은 11pt 아래로 줄이지 않는다. 9.5~10pt는 하위 항목·표, 8.5~9pt는 주석·출처에만 사용한다.
- 글꼴 등록이 불가능하면 사용자 승인 후에만 Noto Sans KR, Apple SD Gothic Neo, Aptos 순으로 대체한다.

## 정보 밀도와 설명 수준

- 처음 보는 독자가 `핵심 결론·판단 / 근거(배경·수행·결과·영향) / 다음 조치`를 순서대로 찾을 수 있게 화면을 구성한다.
- 핵심 결론·판단은 제목 바로 아래에서 현재 상태와 의미를 1~2줄로 연결하고, 주요 영역은 구체 사실과 이유를 2~4개 항목으로 설명한다.
- 진행 경과는 `날짜·단계명` 아래에 활동·변화 한 줄을 두고, 주요 변곡점은 2~4개 상세 항목을 묶어 설명한다.
- 문제·결과 영역은 `번호 또는 항목명 → 구체 현상·근거 → 영향 → 조치 → 확인된 개선 또는 검증 계획`의 순서로 작성한다.
- 적용·완료·개선 조치는 효과가 드러나는 문장으로 쓴다(예: `시스템 프롬프트 강화를 통한 오류율 개선`). 효과가 검증되지 않았으면 기대효과와 검증 계획을 괄호로 밝힌다(예: `시스템 프롬프트 강화(오류율 개선 여부 재검증 예정)`).
- 원문에 예시가 있으면 짧은 예시는 관련 문장 뒤 괄호로, 길거나 여러 개인 예시는 인접한 사례·예시 영역으로 배치한다(예: `예외 문서 재처리(예: 수기 서식·저화질 스캔)`). 예시가 핵심 근거이면 회색 부가 설명으로 축소하지 않는다.
- 큰 빈 공간이 남고 검증된 정보가 있으면 글자나 박스를 키우기 전에 배경·원인·수치·영향·후속 행동 중 빠진 설명을 추가한다.
- 한 불릿은 기본 한 줄, 최대 두 줄로 제한하며 문단을 늘어놓는 대신 상하위 불릿과 짧은 영향 문장으로 압축한다.
- `[항목]`, `[수치]`, `YYYY-MM-DD` 같은 템플릿 자리표시자가 남아 있으면 정보 밀도와 무관하게 미완성으로 판정한다.

## 날짜 표기

- 년·월만 쓰면 `'26.05月`처럼 작은따옴표, 2자리 연도, 점, 2자리 월, 한자 `月`을 사용한다.
- 년·월·일을 쓰면 `2026-05-21`처럼 `YYYY-MM-DD`를 사용한다.
- 월·일만 쓰면 `03/26`처럼 `MM/DD`를 사용한다.
- 한 타임라인·표·목록 안에서는 한 정밀도와 형식으로 통일하고 기간 양 끝도 같은 형식으로 쓴다(예: `'26.03月~'26.05月`, `2026-03-01~2026-05-31`, `03/01~05/31`).
- 원문이 `초·중순·말` 같은 근사 시점만 제공하면 임의의 일자를 만들지 말고 같은 구조로 통일한다(예: `'26.03月 초`, `'26.05月 중순`).

## 항목과 불릿

- 핵심 요약은 검은 사각형 `■` 또는 짧은 문장으로 시작하고 한두 줄 안에 끝낸다.
- 주요 항목은 `1. 항목 제목` 또는 `① 항목 제목`처럼 번호와 Bold 제목으로 시작한다.
- 1단계 본문은 `•` 12pt, 2단계는 `–` 10pt와 한 단계 깊은 행잉 인덴트를 사용한다.
- 불릿 기호를 텍스트에 직접 입력하지 말고 편집 가능한 문단 불릿으로 구현한다.
- 화살표 `→`는 원인→결과, 현재→목표, 판단→조치처럼 실제 흐름이 있을 때만 사용한다.
- 문장 전체를 빨갛게 만들지 말고 실제로 강조해야 하는 구문만 `#C00000`으로 바꾼다.

## 박스, 탭, 표

- 핵심 내용 패널은 흰색 바탕, 1px 회색 테두리, 매우 약한 그림자를 사용한다.
- 패널 제목은 네이비 직사각형 탭에 흰색 Bold로 쓰고 패널 상단 테두리보다 12~18px 위에 겹친다.
- 네이비 탭은 흰색 배경 패치가 아니라 패널 위에 떠 있는 불투명 라벨로 표현한다.
- 일반 서브헤더가 테두리 위에 있을 때는 흰 배경으로 선을 끊지만, 핵심 패널은 네이비 탭을 우선한다.
- `G01` 상위 그룹은 흰색 바탕·1px 회색 테두리·약한 그림자와 네이비 그룹 탭을 한 번만 사용한다. 내부 하위 항목에는 별도 외곽 테두리·그림자·떠 있는 네이비 탭을 반복하지 않는다.
- `G01` 내부 하위 항목은 `1. 항목명`, `2. 항목명` 같은 네이비 번호형 소제목과 1px 회색 세로 구분선으로 나눈다. 하위 항목별 `A01` 회색 한 줄 정리는 같은 하단 기준선에 맞춘다.
- 설명·이슈 묶음은 연회색 라운드 박스를 사용하고 장식용 카드는 만들지 않는다.
- 표 머리글은 연회색 10pt Bold, 본문은 10pt, 행 구분선은 얇은 회색으로 한다.
- 표 위에는 기간, 대상, 종합 결과를 먼저 배치하고 표는 결론을 검증하는 근거로 사용한다.
- 타임라인 날짜는 선택한 표준 형식이 한 줄로 들어가는 전용 열을 확보하고 날짜, 노드, 설명 사이에 10px 이상의 여백을 둔다.
- 번들 템플릿에는 참고자료·부가 설명 자리표시자나 빈 주석 영역을 두지 않는다.

## 강조, 참고, 예시, 시각자료

### 문장과 영역 강조

- 문장 안 핵심 결론·변화·수치만 `#C00000`으로 강조하고 나머지 문장은 기본 검정으로 유지한다.
- 더 강한 강조가 필요한 한 구문에는 글자 자체의 밑줄 대신 구문 폭에 맞춘 `#C00000` 선을 글자 아래 2~4px 지점에 1.5~2pt로 그린다.
- 한 문장에는 빨간 강조 구문 하나, 한 슬라이드에는 강한 빨간 밑줄 하나를 기본 상한으로 한다.
- 박스·표·화면의 특정 영역을 지목할 때는 `#C00000` 1.5~2pt 무채움 사각형으로 6~10px 여백을 두고 감싼다.
- 빨간 영역 사각형은 강조 대상 뒤 레이어에 놓아 글자나 화면을 가리지 않으며, 기존 컨테이너 테두리와 혼동될 만큼 넓게 감싸지 않는다.

### 참고자료·용어와 부가 설명

- 첨부·원문·근거 자료를 참고하거나 어려운 약어·전문용어를 설명해야 하면 `※`로 시작하는 `#3B7D23` 9~10pt 한 문장을 관련 문구 바로 옆에 둔다.
- 용어 주석은 `※ [용어]: [처음 듣는 사람이 이해할 수 있는 쉬운 한 문장 정의]` 형식으로 쓰고, 용어를 용어로 다시 설명하지 않는다.
- 초록 주석은 무엇을 왜 참고해야 하는지 명확히 쓰고, 슬라이드 하단의 출처나 무관한 섹션에 떨어뜨리지 않는다.
- 조건·예외·용어 같은 부가 설명은 `#A6A6A6`로 쓰고 `※` 또는 괄호 안에 넣어 대상보다 1~2pt 작은 크기로 바로 옆에 둔다.
- 초록은 외부·첨부 참고자료와 용어 정의, 회색은 조건·예외 같은 보조 설명으로 구분하며 같은 내용을 두 색으로 중복하지 않는다.

### 사례·예시

- 짧은 예시는 관련 문장과 같은 텍스트 상자 안에서 `(예: …)`로 붙여 개념과 사례를 함께 읽게 한다.
- 길거나 여러 개인 예시는 주장 바로 옆의 사례·예시 영역에 두고 사례명, 확인 내용, 의미 또는 결과를 함께 적는다.
- 예시가 결론을 뒷받침하는 핵심 근거이면 본문과 같은 위계로 유지하고, 단순 보조 예시일 때만 작은 회색 괄호 문장으로 처리한다.
- 원문에 없는 사례나 성과는 만들지 않는다.

### 도식·flowchart·화면 캡처 영역

- 도식·flowchart·화면 캡처가 이해를 실질적으로 높이거나 사용자가 자료를 제공할 예정이면 본문 그리드 안에 편집 가능한 사각형 영역을 먼저 확보한다.
- 실제 자료가 없을 때는 흰색 또는 매우 연한 회색 바탕, 1px 회색 테두리의 사각형에 `[도식: 설명]`, `[Flowchart: 설명]`, `[화면 캡처: 설명]`처럼 자료 유형과 역할을 표시한다.
- 자리 영역은 슬라이드의 25~45%를 권장하며, 주변 문구를 과도하게 축소해야 한다면 백데이터 슬라이드로 분리한다.
- 시각자료 자리 사각형은 일반 회색 테두리를 사용한다. 그 안의 특정 부분을 강조할 때만 별도의 빨간 무채움 사각형을 겹친다.

## 컴포넌트 조합 시각 규칙

- 상황별 구성과 필수 슬롯은 `component-system.md`의 조합 레시피를 따른다.
- 공통 크롬은 한 번만 사용하고 화면에서 가장 중요한 주 구성 하나를 먼저 보이게 한다.
- 컴포넌트는 독립 UI 카드처럼 흩어놓지 말고 보고 논리에 따라 하나의 평면적 구성으로 연결한다.
- 동일 비중의 두 패널은 크기, 제목 기준선, 회색 한 줄 정리 위치를 맞춘다.
- 타임라인·요약 같은 상위 영역과 관련 항목 두 개를 `1:2`로 배치할 때는 왼쪽 32~36%, 오른쪽 `G01` 64~68%의 최상위 두 프레임만 만든다. `G01` 내부는 좌우 1:1과 20~28px 안쪽 여백, 1px 세로 구분선을 사용한다.
- `G01`의 그룹 탭은 오른쪽 영역 전체의 공통 의미를 말하고, 하위 소제목은 항목 차이만 말한다. 그룹 탭과 하위 소제목에 같은 크기·색상·그림자 효과를 반복하지 않는다.
- 두 하위 항목이 독립 선택지이거나 슬라이드 전체의 최상위 핵심 내용일 때만 `R02`의 두 독립 패널을 사용한다. 다른 상위 영역과 함께 놓인 관련 항목은 `G01`로 묶는다.
- 단일 개선 과제는 왼쪽 약 45%에 과제 밴드·`AS-IS → TO-BE`·효과·일정을 두고, 오른쪽 약 55%에 실제 과제 세부 내용을 둔다.
- 오른쪽 설명 영역은 상단에 3단계 수행 흐름과 단계별 한 줄 설명, 하단에 `입력·대상 / 처리·분석 / 산출·활용` 3개 구성축과 최종 산출·운영 결과를 배치한다.
- 세부 설명은 작은 카드를 반복하기보다 단계 흐름과 구성축 두 레벨로 제한하고, 한 항목이 두 줄을 넘으면 축약하거나 백데이터로 옮긴다.
- 복수 개선 과제는 두 개의 독립 카드 대신 하나의 큰 패널 안에 두 행으로 놓고 `과제 → 현재 → 목표 → 효과` 기준선을 맞춘다.
- `현재 방식`과 `문제·제약`, `목표 방식`과 `변경 사항·운영 기준`을 각각 제목과 세부 불릿으로 분리한다. 두 문구를 작은 색상 박스 하나에 압축하지 않는다.
- 공통 추진은 실제 공통 단계가 있을 때만 네이비 라벨과 가는 트랙·노드로 표현하고, 단계별 큰 박스를 반복하지 않는다.
- 화살표는 전환 흐름 안에서만 `→` 텍스트로 사용하고 독립적인 네이티브 화살표 도형은 기본값으로 사용하지 않는다.
- 웹용 범용 템플릿은 직사각형을 기본 도형으로 사용한다. 라운드 박스나 특수 도형은 해당 장에서 렌더링이 검증된 경우에만 쓴다.
- `TO-BE`는 괄호 없이 쓰고 10~12pt 기준 최소 96px 폭을 확보해 대체 글꼴과 웹 미리보기에서도 한 줄을 유지한다.
- 날짜는 선택한 표준 형식의 실제 글자 폭보다 최소 20% 넓은 전용 텍스트 영역을 확보한다.
- 회색 과제 밴드는 단일 과제 장표, 회색 한 줄 정리는 2개 핵심 내용 장표, 회색 해석 바는 백데이터 장표에서만 사용한다.
- 주·보조 관계가 있을 때만 열 너비를 달리하고, 장식 목적의 비대칭은 만들지 않는다.
- 네이비 탭, 회색 정리 영역, 빨간 결론 바를 한 화면에 모두 반복 사용하지 않는다.
- 여백이 부족하면 컴포넌트를 축소하거나 글자를 줄이지 말고 선택 요소를 제거하거나 백데이터로 분리한다.
- 백데이터 제목에는 `- 백데이터`를 붙이고, 표·차트 아래 해석은 핵심 변화나 한계만 강조한다.

## 검수 기준

- 제목과 빨간 구분선의 위치가 모든 슬라이드에서 일관적이고 제목에 불릿 표식이 없는지 확인한다.
- 제목 바로 아래 첫 본문이 핵심 결론·판단이며 `보고 목적` 라벨이 남지 않았는지 확인한다.
- 타임라인 날짜가 노드나 오른쪽 설명과 겹치지 않는지 확인한다.
- 반복 날짜가 `'26.05月`·`2026-05-21`·`03/26` 중 내용 정밀도에 맞는 한 형식으로 통일됐는지 확인한다.
- 모든 `TO-BE` 라벨이 웹 미리보기에서도 한 줄인지, 실제 첨언이 없을 때 녹색 문구가 남지 않는지 확인한다.
- 처음 보는 독자가 배경·수행·결과·영향·다음 조치를 찾을 수 있고 각 결론의 이유를 재진술할 수 있는지 확인한다.
- 큰 빈 공간이 있는데도 검증된 근거·원인·영향·조치가 누락되지 않았는지 확인한다.
- 적용·완료 조치가 확인된 개선 결과 또는 검증 계획과 연결됐는지 확인한다.
- 원문 예시가 관련 문장·사례 영역·백데이터에 포함되고 어떤 주장을 뒷받침하는지 분명한지 확인한다.

- 초록 참고자료와 회색 부가 설명이 각각 대상 바로 옆에 있으며 무엇을 설명하는지 모호하지 않은지 확인한다.

- 빨간 영역 강조 사각형이 무채움이며 대상 콘텐츠를 가리지 않는지 확인한다.
