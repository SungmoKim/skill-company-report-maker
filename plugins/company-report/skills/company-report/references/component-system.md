# Component System

보고 상황에 맞는 의미 기반 컴포넌트를 조합해 1페이지 보고서를 설계한다. 컴포넌트 ID는 제작 과정의 공통 언어이며 슬라이드 화면에는 노출하지 않는다.

## 목차

1. 역할과 우선순위
2. 선택 순서
3. 컴포넌트 카탈로그
4. 시각 계층 판단
5. 조합 레시피
6. 레시피별 화면 소유권
7. 콘텐츠 수용량
8. 구현과 검수

## 역할과 우선순위

- `report-types.md`는 보고 상황과 내용의 순서를 정한다.
- 이 문서는 필요한 컴포넌트와 조합 방식을 정한다.
- `design-system.md`는 선택된 컴포넌트의 색상, 글꼴, 간격, 선을 정한다.
- `assets/template.pptx`는 대표 조합의 완성 형태와 시각적 기준을 제공한다.
- 사용자 회사 템플릿이 있으면 그 템플릿의 마스터와 레이아웃 제약을 우선한다.
- 컴포넌트 조합이 템플릿 전체 슬라이드 복제보다 내용에 더 적합하면 같은 시각 문법으로 새 구성을 만든다.

## 선택 순서

1. 보고 대상, 청중이 판단해야 할 질문, 핵심 결론을 확인한다.
2. `report-types.md`에서 보고 상황과 권장 레시피를 고른다.
3. 레시피의 필수 컴포넌트만 먼저 배치한다.
4. 최상위 영역을 먼저 세고, 같은 질문·영향·결론을 공유하는 하위 항목은 `G01` 안에 묶는다.
5. 근거가 있을 때만 선택 컴포넌트를 추가한다.
6. 한 화면에 맞지 않으면 컴포넌트를 축소하지 말고 내용을 줄이거나 백데이터로 분리한다.
7. 구조 제안에 선택한 레시피, 최상위 영역, 하위 그룹과 주요 컴포넌트를 텍스트로 밝히고 승인받는다.

## 컴포넌트 카탈로그

| ID | 컴포넌트 | 사용하는 상황 | 필수 슬롯 | 주요 제한 |
| --- | --- | --- | --- | --- |
| `C01` | 문서 크롬 | 모든 슬라이드 | 문서명, 빨간 구분선, 푸터, 페이지 | 제목에는 불릿을 붙이지 않는다. |
| `C02` | 핵심 결론·판단 | 모든 본문 장표에서 결론·상태·권고를 먼저 말할 때 | 결론과 근거·의미·다음 방향 | 제목 바로 아래 첫 본문에 1~2줄로 두며 `보고 목적` 라벨을 사용하지 않는다. |
| `C03` | 섹션 라벨 | 영역의 역할을 구분할 때 | 짧은 명사형 라벨 | 네이비 탭 또는 세로 라벨 중 하나만 쓴다. |
| `C04` | 과제 헤더 | 단일·복수 과제의 이름과 목표를 밝힐 때 | 과제명, 목표 시점, 기대효과 | 단일은 상단 중앙 밴드, 복수는 행 왼쪽에 정렬한다. |
| `G01` | 상위 그룹 프레임 | 다른 최상위 영역과 나란히 놓인 2~3개 관련 항목을 하나의 의미 단위로 묶을 때 | 그룹명, 공통 질문·판단, 하위 항목, 공통 시사점(선택) | 외곽 프레임·그룹 탭·그림자는 한 번만 사용하고 하위 항목에 독립 외곽 패널·그림자를 반복하지 않는다. |
| `H01` | 문장 강조 | 문장 안 핵심 결론·변화·수치를 강조할 때 | 강조 구문 | `#C00000`, 가장 강한 한 구문만 별도 빨간 밑줄을 허용한다. |
| `H02` | 영역 강조 | 기존 박스·표·화면의 특정 범위를 지목할 때 | 강조 대상 영역 | `#C00000` 무채움 사각형으로 감싸고 내용 위를 가리지 않는다. |
| `N01` | 참고자료·용어 주석 | 첨부·원문을 보거나 어려운 용어를 이해해야 할 때 | `※`와 참고 내용 또는 쉬운 정의 | `#3B7D23`, 대상 바로 옆에 한 문장으로 두며 템플릿 자리표시자는 만들지 않는다. |
| `N02` | 부가 설명 | 조건·예외 등 보조 맥락이 필요할 때 | `※` 또는 괄호 설명 | `#A6A6A6`, 대상보다 작은 글씨로 바로 옆에 둔다. |
| `E01` | 진행 타임라인 | 사건·검증·업무 경과 | 날짜, 사건명, 활동과 변화 | 3~5개 사건, 날짜 전용 열을 확보하고 `'26.05月`·`2026-05-21`·`03/26` 중 한 형식으로 통일한다. |
| `E02` | 핵심 수치 | 목표·실적·변화량 강조 | 수치, 단위, 기간, 기준 | 수치마다 의미와 기준을 붙인다. |
| `E03` | 결과·이슈 표 | 여러 항목을 같은 기준으로 검증 | 항목, 확인 내용, 조치·개선·한계 | 3~5행, 조치만 쓰지 말고 확인된 개선 또는 검증 계획을 연결한다. |
| `E04` | 핵심 내용 블록 | 동등한 현황·효과·발견·근거 | 제목, 설명·근거, 선택 탭 | 기본 2개, 최대 3개까지만 병렬 배치한다. |
| `E05` | 시각자료·근거 영역 | 도식·flowchart·화면 캡처·문서 예시가 필요할 때 | 자료 유형, 목적, 출처 또는 추가할 내용 | 자료가 없으면 편집 가능한 사각형으로 영역을 예약하고 장식용 이미지는 넣지 않는다. |
| `E06` | 근거 차트 | 추이·비교·구성 데이터를 설명할 때 | 지표, 단위, 기간, 출처 | 결론을 검증하는 시리즈만 남긴다. |
| `E07` | 세부 과제 설명 | 단일 과제의 실제 수행 내용을 설명할 때 | 단계별 활동, 입력·처리·산출, 운영 방식 | 3단계와 3개 구성축을 기본으로 하고 근거 없는 슬롯은 억지로 채우지 않는다. |
| `E08` | 사례·예시 | 구체 사례가 주장이나 개념의 이해를 높일 때 | 연결할 주장, 실제 예시, 의미 또는 결과 | 짧은 예시는 본문 괄호로 쓰고, 길거나 여러 개인 예시만 별도 영역에 두며 근거에 없는 사례를 만들지 않는다. |
| `T01` | AS-IS→TO-BE | 업무 개선 전후를 설명할 때 | 현재 방식, 문제·제약, 목표 방식, 변경 사항 | 현재·목표의 제목과 세부 항목을 분리하고 실제 변화가 있을 때만 화살표를 쓴다. |
| `T02` | 단계형 흐름 | 추진·확장·실행 단계 | 단계명, 핵심 활동 | 3~5단계, 단계별 문구 길이를 맞춘다. |
| `T03` | 범위 확장 | 일부→핵심→전체 같은 적용 확대 | 단계, 대상 범위 | 단계별 범위가 커지는 논리가 보여야 한다. |
| `A01` | 개별 한 줄 정리 | 패널별 의미를 압축할 때 | 시사점 또는 기대효과 | 회색 영역 한 문장, 본문을 반복하지 않는다. |
| `A02` | 종합 정리·목표 | 여러 근거의 합산 결론이 있을 때 | 종합 결론 또는 목표 | 합산 결론이 없으면 생략한다. |
| `A03` | 결정·요청 | 승인·선택·지원이 필요할 때 | 요청 내용, 기한, 후속 행동 | 요청 주체와 행동이 명확해야 한다. |
| `B01` | 백데이터 해석 | 표·차트의 의미를 정리할 때 | 출처, 기준일, 해석 | 표 아래에서 결정에 미치는 영향까지 쓴다. |

`E04`는 분리가 중요한 `패널형`, 한 화면의 흐름을 유지하는 `평면형`, `G01` 안에서 쓰는 `하위 블록형`으로 나눈다. `R02`·`R05`는 패널형, `R04`는 평면형을 기본으로 한다. 하위 블록형은 번호형 소제목·내부 구분선·정렬된 `A01`만 사용하며 독립 그림자와 떠 있는 네이비 탭을 쓰지 않는다. `H01`·`H02`·`N01`·`N02`·`E05`·`E08`은 근거와 필요성이 있을 때 어떤 레시피에도 선택적으로 추가한다.

## 시각 계층 판단

1. 공통 크롬을 제외한 최상위 의미 영역을 먼저 센다.
2. 두 항목이 같은 상위 질문에 답하거나 같은 업무 영향·결론으로 합쳐지면 형제 항목이며 `G01`의 자식으로 둔다.
3. `1:2` 혼합 구성은 `왼쪽 상위 영역 1개 : 오른쪽 G01 1개`로 구현한다. 오른쪽의 두 자식을 독립 외곽 패널로 만들어 세 개의 동급 카드처럼 보이게 하지 않는다.
4. 외곽 테두리·그림자·큰 네이비 탭은 최상위 영역만 소유한다. 하위 항목은 번호형 소제목과 세로 구분선으로 구분한다.

## 조합 레시피

### `R01` 결과·이슈

- 조합: `C01 + C02 + E01 + E03 + A03`
- 진행 경과보다 판단이 중요하면 `E01`을 줄이고 `E02`를 추가한다.
- 각 경과에는 활동·변화 한 줄을, 각 이슈에는 현상·원인·영향·조치·확인된 개선 중 최소 세 가지를 담아 처음 보는 청중도 판단 근거를 따라갈 수 있게 한다.
- 번들 템플릿 기준: 1번 슬라이드

### `R01-G2` 그룹형 결과·이슈

- 조합: `C01 + C02 + E01 + G01[(E04 하위 블록형 + A01) × 2] + A03`
- 진행 경과와 정확히 두 개의 상세 이슈가 `왜 현업 적용이 어려운가`처럼 하나의 공통 판단을 뒷받침할 때 사용한다.
- 화면의 최상위 영역은 `E01`과 `G01` 두 개다. 기본 너비는 `1:2`이고, `G01` 안의 두 하위 블록은 `1:1`로 나눈다.
- `G01`에는 공통 그룹명과 필요할 때 공통 판단 한 줄을 두고, 각 하위 블록에는 현상·적용 조치·확인된 개선 또는 검증 계획·남은 한계와 `A01` 업무 영향을 같은 기준선으로 배치한다.
- 번들 템플릿 1번 슬라이드의 상위 2단 구조와 5번 슬라이드의 하위 내용 구성을 함께 참고하되, 5번의 독립 외곽 패널·그림자는 하위 블록에 복제하지 않는다.

### `R02` 2개 핵심 내용

- 조합: `C01 + C02 + (E04 + A01) × 2 + A02(선택)`
- 두 현황·효과·발견을 같은 비중으로 설명한다. 독립적인 내용이면 `A02`를 생략한다.
- 번들 템플릿 기준: 5번 슬라이드

### `R03` 업무 개선·과제 수 대응

- 단일 과제 `R03-S`: `C01 + C02 + T01 + A01 + E07 + T02(선택)`
- 복수 과제 `R03-M`: `C01 + C02 + (C04 + T01 + A01) × 2 + T02(선택)`
- 과제가 하나면 문서명에서 과제명을 밝히고 그 아래 `C02`에서 개선 결론을 먼저 제시한다. 왼쪽에는 `AS-IS→TO-BE`·효과·일정을 요약하고 오른쪽에는 실제 수행 단계와 입력·처리·산출을 설명한다. 두 번째 빈 과제 영역은 만들지 않는다.
- 과제가 둘이면 두 개의 평면형 행에 과제 헤더, 현재, 목표, 효과를 같은 기준선으로 맞춘다.
- 적용·완료한 변경 사항은 기대효과만 반복하지 말고 확인된 개선 결과 또는 이후 검증 계획과 연결한다.
- `T02`는 실제로 공유하는 추진 단계가 있을 때만 한 줄 트랙으로 둔다. 공유하지 않으면 과제별 다음 조치로 바꾼다.
- 번들 템플릿 기준: 단일 2번, 복수 3번 슬라이드

### `R04` 확장 적용·전략 계획

- 조합: `C01 + C02 + C03 + T03 + E04 × 2 + E05(선택) + A02`
- 사업·구조처럼 관점이 다르면 세로 라벨로 구획하고 최종 목표는 한 문장으로 합친다.
- 번들 템플릿 기준: 4번 슬라이드

### `R05` 의사결정

- 조합: `C01 + C02 + (E04 + A01) × 2 + A03`
- `E04` 탭은 선택지명, `A01`은 선택지별 판단 요약, `A03`은 권고안과 결정 기한으로 바꾼다.
- 번들 템플릿 기준: 5번 슬라이드의 의사결정 변형

### `R06` 백데이터

- 조합: `C01 + C02 + (E03 또는 E06) + B01`
- 표·차트는 요약 슬라이드의 결론 하나를 검증해야 하며 새로운 결론을 추가하지 않는다.
- 번들 템플릿 기준: 6번 슬라이드

## 레시피별 화면 소유권

- `R01`은 진행 타임라인과 결과·이슈 표만 사용한다. 전환 화살표나 공통 추진 트랙을 섞지 않는다.
- `R01-G2`는 진행 타임라인과 하나의 `G01`만 최상위 영역으로 사용한다. `G01` 안의 두 `E04`는 하위 블록이며 별도 외곽 패널·그림자·큰 탭을 소유하지 않는다.
- `R03-S`의 주 구성은 상단의 `C02`, 왼쪽 요약 영역의 `AS-IS→TO-BE`·효과·선택적 추진 트랙과 오른쪽 설명 영역의 `E07`로 제한한다.
- `R03-M`은 두 과제 행과 실제로 공유하는 경우의 공통 추진 트랙만 사용한다.
- `R04`는 상단의 `C02`, 관점 세로 라벨, 범위 단계, 추진 방향, 목표 영역으로 구성한다. 단일 과제 밴드나 회색 요약 카드를 올리지 않는다.
- `R02`는 두 핵심 패널과 패널별 한 줄 정리, 선택적 종합 정리만 사용한다. 전환 화살표와 타임라인 노드를 두지 않는다.
- `R06`은 근거 표·차트와 출처·해석 영역만 사용한다. 다른 레시피의 노드·화살표·과제 밴드를 두지 않는다.
- `H01`·`H02`·`N01`·`N02`는 실제 대상 가까이에 겹쳐 쓰는 선택 강조·주석이며, `E05`는 선택한 레시피의 한 영역을 대체할 때만 사용한다.
- 한 슬라이드의 객체 이름은 슬라이드 번호와 레시피를 포함해 덱 전체에서 고유하게 관리한다.

## 콘텐츠 수용량

- 공통 크롬을 제외하고 한 슬라이드에는 하나의 주 구성과 서로 다른 질문에 답하는 2~4개 보조 영역을 둔다.
- 동일한 모양의 패널을 반복해 대시보드처럼 만들지 말고 논리적으로 묶이는 요소만 한 패널에 넣는다.
- `G01`은 하위 항목 2개를 기본으로 하고 최대 3개까지만 허용한다. 각 하위 항목은 제목 1개, 설명 2~4개, `A01` 한 문장으로 제한한다.
- 각 주요 영역에는 제목 1개, 핵심 사실 1개, 원인·근거·영향·조치 중 2~4개 설명 항목을 권장한다.
- `A01`은 한 문장, `A02`와 `A03`은 한두 줄을 넘기지 않는다.
- `H01`은 문장당 한 구문, `H02`는 슬라이드당 한 영역을 기본으로 하며 빨간색을 장식처럼 반복하지 않는다.
- `N01`은 참고자료 또는 어려운 용어별 한 문장, 슬라이드당 두 개를 기본 상한으로 한다. 더 많으면 본문을 쉬운 말로 바꾸고 `N02`는 한 문장만 둔다.
- `E05` 자리에는 자료 유형과 역할을 짧게 표시하고, 구조 승인 단계에서 필요한 면적과 실제 자료 제공 여부를 확인한다.
- `E08`은 짧은 예시 한두 개를 기본으로 하고, 관련 주장 바로 옆에서 괄호 또는 한 줄 사례로 연결한다. 예시가 많으면 대표 사례는 본문에 두고 나머지는 백데이터로 옮긴다.
- 두 열은 같은 비중이면 1:1로, 주·보조 관계가 분명할 때만 최대 2:1로 조정한다.
- 복수 과제 행은 정보를 작은 카드로 다시 쪼개지 말고 `과제 → 현재 → 목표 → 효과`가 한 방향으로 읽히게 한다.
- `E07`은 수행 단계 3개를 각 1줄로 설명하고, 입력·대상 / 처리·분석 / 산출·활용의 각 축에는 최대 3개 항목만 둔다.
- 단일 과제의 세부 내용이 `E07` 수용량을 넘으면 핵심 작동 원리와 최종 산출만 남기고 업무 흐름·화면·상세 요건은 백데이터로 분리한다.
- 검증된 정보가 있는데 본문에 큰 빈 공간이 남으면 `배경·수행·결과·영향·다음 조치` 중 빠진 설명을 먼저 보강한다.
- 컴포넌트가 수용량을 넘으면 본문 축약→선택 컴포넌트 제거→백데이터 분리 순으로 해결한다.

## 구현과 검수

- 현재 작업의 편집 가능한 슬라이드 소스 안에서 반복 요소를 로컬 헬퍼로 구현할 수 있다.
- 저장소에 별도 렌더러, 컴포넌트 엔진, npm 패키지 또는 자동화 스크립트를 추가하지 않는다.
- 모든 컴포넌트는 네이티브 텍스트, 도형, 표, 차트로 구현하고 그룹 이미지로 평면화하지 않는다.
- `G01`은 외곽 프레임과 그룹 헤더를 먼저 만들고, 그 안에 하위 블록과 내부 구분선을 배치한다. 객체 이름에도 `group`과 `child-1`, `child-2` 관계를 구분한다.
- `H01`의 강한 밑줄은 강조 구문 폭에 맞춘 별도 선, `H02`와 빈 `E05`는 편집 가능한 무채움 사각형으로 구현한다.
- 짧은 예시는 관련 문장과 같은 텍스트 상자에 괄호로 넣고, 별도 `E08`은 연결 대상 제목이나 근거를 함께 표시해 고립된 장식 카드가 되지 않게 한다.
- 웹 프리뷰에서 다른 장의 객체가 섞이면 객체 ID·이름 충돌과 레이어 잔존을 확인하고, 선택한 레시피의 객체만 남긴 상태로 전체 덱을 다시 내보낸다.
- 구조 승인 후에도 같은 소스와 같은 컴포넌트를 수정해 PNG와 PPTX를 만든다.
- 컴포넌트 이름보다 실제 메시지 위계가 먼저 보이는지 확인한다.
- `R01-G2`에서 최상위 외곽 프레임과 그림자가 정확히 두 개이고, 오른쪽 두 항목이 한 그룹 헤더·외곽 프레임 안에 완전히 포함되는지 확인한다.
- 모든 자리표시자가 실제 문구로 바뀌었고 청중이 사전 설명 없이도 주요 판단의 이유를 재진술할 수 있는지 확인한다.
- 제목 아래 첫 본문이 핵심 결론·판단이며 `보고 목적` 라벨이 남지 않았는지 확인한다.
- 반복 날짜가 `'26.05月`·`2026-05-21`·`03/26` 중 한 정밀도와 형식으로 통일됐는지 확인한다.
- 적용·완료 조치에 확인된 개선 결과 또는 검증 계획이 연결되고, 원문 예시가 괄호·`E08`·백데이터 중 적절한 위치에 포함됐는지 확인한다.
- 의도하지 않은 겹침, 잘림, 줄바꿈과 컴포넌트 간 최소 여백을 전체 크기 PNG에서 확인한다.
