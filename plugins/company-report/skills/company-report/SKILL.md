# Company Report

회사 보고서를 기본 1페이지 요약으로 만들고, 승인된 PNG와 같은 편집 가능한 원본에서 PPTX를 내보내라.

## 절대 규칙

- 아래 1~6단계를 순서대로 수행하라.
- 구조 승인과 PNG 승인이라는 두 승인 게이트를 생략하지 마라.
- 한 슬라이드에는 하나의 핵심 메시지만 두어라.
- 표지를 제외한 모든 장표는 `문서명 → 핵심 결론·판단 → 근거·세부 → 다음 조치` 순서의 두괄식으로 구성하라.
- 제목 아래 첫 본문 영역에 한 문장 결론을 두고, 화면 라벨이나 문구에 `보고 목적` 또는 `[보고 목적]`을 사용하지 마라.
- 기본 비율을 16:9로 유지하라.
- PNG와 PPTX를 하나의 편집 가능한 슬라이드 소스에서 생성하라.
- 첫 렌더링 PNG를 사용자에게 바로 제시하지 말고, 1280×720 Vision QA를 통과한 PNG만 제시하라.
- 승인 전 PNG를 최종 PPTX에 이미지로 삽입하는 방식을 사용하지 마라.
- 사용자가 PNG를 명시적으로 승인하기 전에는 최종 PPTX를 전달하지 마라.
- 근거가 없는 수치, 사실, 인용, 성과를 만들지 마라.
- 실제 회사 스크린샷, 고객 정보, 사내 화면, 개인정보를 스킬 저장소에 복사하지 마라.

## 리소스와 도구

- 요구사항을 받은 뒤 보고 유형을 고를 때 [report-types.md](references/report-types.md)를 읽어라.
- 구조 제안 전에 [component-system.md](references/component-system.md)에서 조합 레시피를 골라라.
- 구조 승인 후 레이아웃 작성 전에 [design-system.md](references/design-system.md)로 시각 규칙을 적용하라.
- 슬라이드 소스를 만들기 전에 [font-setup.md](references/font-setup.md)를 읽고 번들 글꼴을 등록·검증하라.
- 사용자 회사 템플릿이 있으면 그것을 최우선으로 사용하라.
- 회사 템플릿이 없으면 [template.pptx](assets/template.pptx)의 시각 문법과 선택한 레시피 장표만 참고하고, 템플릿 문구나 전체 장표를 그대로 복제하지 마라.
- 호스트 환경이 제공하는 프레젠테이션 제작·렌더링·검증 기능을 사용하라.
- `Presentations` 스킬이 제공되면 읽고 그 기능을 우선 사용하라.
- npm, Sharp, 자체 SVG 렌더러, 자체 렌더링 엔진을 추가하지 마라.
- 별도 검사기나 자동화 스크립트를 만들지 말고 프레젠테이션 스킬의 기능을 우선하라.

## 실행 환경

- ChatGPT 웹 Work 모드, ChatGPT 데스크톱 Work/Codex, Codex CLI에서 같은 승인 흐름을 적용하라.
- 로컬 저장소, 로컬 셸, 사용자 PC의 절대 경로가 있다고 가정하지 마라.
- 레퍼런스와 번들 템플릿은 항상 이 스킬 디렉터리 기준의 상대 경로로 찾아라.
- 사용자 회사 템플릿과 근거 자료는 현재 대화에 업로드된 첨부 파일에서 받아라.
- 호스트가 제공하는 임시 작업공간에서 편집 가능한 슬라이드 소스와 렌더링 결과를 유지하라.
- 웹 호스트에서는 시스템 영역이 아닌 현재 작업 세션의 사용자 글꼴 디렉터리에만 번들 글꼴을 등록하라.
- 글꼴 등록 또는 확인 기능이 없으면 대체 글꼴을 임의로 사용하지 말고 제한을 사용자에게 알려라.
- PNG와 PPTX는 로컬 경로만 알려주지 말고 사용자가 대화에서 열 수 있는 첨부 파일로 제공하라.
- 호스트에 프레젠테이션 생성 또는 렌더링 기능이 없으면 성공한 것처럼 가장하지 말고 제한과 가능한 다음 단계를 알려라.

## 1. 요구사항 정리

사용자가 이미 준 정보는 다시 묻지 말고 다음 항목으로 정리하라.

1. 보고 대상
2. 핵심 결론 또는 권고안
3. 현재 상황과 청중이 판단해야 할 질문
4. 생각하고 있는 보고 방향 및 흐름(선택)
5. 근거 자료와 출처
6. 원하는 분량
7. 회사 템플릿 또는 디자인 기준

필수 정보가 없으면 결과가 크게 달라지는 질문만 짧게 물어라.
사용자가 판단을 맡기면 합리적인 가정을 명시하고 계속하라.
민감 자료는 현재 작업에만 사용하고 저장소 자산이나 공개 예시에 포함하지 마라.
정리 결과를 짧은 브리프로 보여준 뒤 구조 제안으로 이어가라.

## 2. 보고서 구조 제안과 승인

보고 상황을 다음 중 하나로 분류하라.

- 신규 기획·제안
- 정기 진행 보고
- 문제·이슈 보고
- 결과·성과 보고
- 업무 개선·단일 또는 복수 과제 보고
- 의사결정 요청

기본안은 반드시 1페이지 요약 1장으로 제안하라.
기본 1페이지에는 표지를 추가하지 말고 문서명 제목에서 바로 시작하라.
보고 유형과 실제 과제 수에 맞는 컴포넌트 조합 레시피를 선택하라. 단일 과제에 빈 비교 영역을 만들지 말고, 선택지 비교는 2개 핵심 내용 레시피를 의사결정 용도로 변형하라.
구조안에서 최상위 영역과 하위 항목을 구분하라. 타임라인·요약 같은 한 영역과, 같은 질문·영향·결론을 공유하는 2~3개 항목이 다른 한 영역에 함께 놓이면 해당 항목을 독립 최상위 패널로 승격하지 말고 `G01` 상위 그룹 안에 묶어라. `1:2` 레이아웃은 최상위 영역 두 개를 뜻하며 동일 위계의 카드 세 개로 만들지 마라.
상세 산식, 원자료, 리스크 목록, 일정, 비교표가 결론을 방해할 때만 백데이터를 제안하라.
백데이터가 필요하면 장수와 필요한 이유를 함께 적어라.

구조안에는 각 슬라이드별로 다음을 텍스트로 제시하라.

- 슬라이드 또는 섹션명
- 제목 아래 가장 먼저 제시할 핵심 결론·판단
- 섹션별 전달 역할과 핵심 내용
- 전체 흐름에서의 역할과 사용할 조합 레시피·핵심 컴포넌트
- 최상위 영역과 그 안에 포함되는 하위 그룹·항목

마지막에 “이 구조로 문구와 레이아웃을 제작해도 될까요?”처럼 명시적 승인을 요청하라.
사용자가 승인·확정·진행 의사를 분명히 밝히기 전에는 문구와 슬라이드 소스를 만들지 마라.
수정 요청을 받으면 구조만 고쳐 다시 승인받아라.

## 3. 문구와 레이아웃 작성

승인된 구조를 기준으로 보고서 제목과 청중이 이해해야 할 한 문장 결론을 구분해 작성하라.
상단 제목은 서술형 결론이 아니라 업무명·과제명·계획명처럼 문서를 식별하는 짧은 명사형 제목으로 써라.
빨간 구분선 아래 첫 본문에는 결론 또는 판단을 한 문장으로 먼저 배치하라. 라벨이 필요하면 `핵심 결론` 또는 `핵심 판단`을 사용하고 `보고 목적`은 사용하지 마라.
처음 듣는 청중을 기준으로 1페이지 안에서 `왜 시작했는가 → 무엇을 했는가 → 무엇이 확인됐는가 → 왜 중요한가 → 무엇을 할 것인가`가 이어지게 하고, 단일 과제는 대상·단계·처리·산출·운영까지 구체적으로 설명하라.
섹션 이름이나 키워드만 나열하지 말고 각 핵심 항목에 주체·대상·방식·근거·영향 중 필요한 사실을 실제 문장으로 작성하라.
적용·완료·개선한 내용을 쓰면 조치명만 적지 말고 확인된 개선 결과를 같은 문장이나 바로 인접한 효과 영역에 연결하라(예: `시스템 프롬프트 강화를 통한 오류율 개선`). 개선이 아직 검증되지 않았으면 성과를 만들지 말고 기대효과와 검증 계획을 적어라(예: `시스템 프롬프트 강화(오류율 개선 여부 재검증 예정)`).
근거 자료에 구체 예시가 있으면 생략하지 마라. 짧은 예시는 관련 문장에 괄호로 붙이고(예: `문서 유형별 예외 처리 적용(예: 수기 서식·저화질 스캔)`), 길거나 여러 개인 예시는 `E08` 사례·예시 영역 또는 백데이터로 분리하되 어떤 주장과 연결되는지 표시하라. 근거에 없는 예시는 만들지 마라.
모든 수치에 단위·기간·기준을 붙이고 필요하면 출처를 각주로 표시하라.
날짜는 정보의 정밀도에 따라 다음 세 형식만 사용하라.

- 년·월: `'26.05月`처럼 작은따옴표와 2자리 연도, 점, 2자리 월, 한자 `月`을 쓴다.
- 년·월·일: `2026-05-21`처럼 `YYYY-MM-DD`로 쓴다.
- 월·일: `03/26`처럼 `MM/DD`로 쓴다.

같은 타임라인·표·목록에서 날짜가 반복되면 가장 적합한 한 정밀도와 형식을 골라 전체 항목에 통일하라. 기간도 양 끝의 형식과 정밀도를 맞춰 쓰고, 원문이 `초·중순·말`처럼 근사 시점만 제공하면 그 표현은 유지하되 반복 항목을 같은 구조로 맞춰라(예: `'26.03月 초`, `'26.05月 중순`).
문장 안 핵심 구문은 `#C00000`으로 강조하고 더 강한 강조 한 곳만 같은 색의 별도 밑줄을 그어라. 기존 영역을 지목할 때는 같은 색의 무채움 사각형으로 감싸라.
첨부·원문 참고자료와 어려운 약어·전문용어 정의는 대상 바로 옆에 `※`로 시작하는 `#3B7D23`의 쉬운 한 문장으로, 그 밖의 부가 설명은 `#A6A6A6`의 더 작은 `※` 또는 괄호 문장으로 두어라.
도식·flowchart·화면 캡처가 필요하지만 자료가 없으면 자료 유형과 역할을 적은 편집 가능한 사각형 영역을 확보하라.
검증된 근거가 있고 큰 빈 공간이 남으면 배경·원인·영향·조치를 보강하고, 그래도 공간이 부족할 때만 문구를 축약하거나 백데이터로 옮겨라.

승인된 조합 레시피의 필수 컴포넌트부터 배치하고 문구와 레이아웃을 하나의 편집 가능한 슬라이드 소스로 구현하라.
`G01`을 사용하는 경우 외곽 프레임·그룹 헤더·그림자는 상위 그룹에 한 번만 적용하고, 하위 항목은 내부 구분선과 번호형 소제목으로 나눠 같은 그룹의 자식으로 보이게 하라.
텍스트, 도형, 표, 차트는 가능한 한 네이티브 편집 객체로 유지하고 객체 이름은 슬라이드·레시피별로 고유하게 하라.
사용자 템플릿의 마스터, 로고, 색상, 여백, 페이지 표기를 보존하라.
사용자 템플릿이 없으면 번들 템플릿의 구성을 적용하되 모든 의미 슬롯을 승인된 실제 문구로 교체하고 `[항목]`, `YYYY-MM-DD` 같은 미해결 자리표시자가 하나라도 남은 시안을 제시하지 마라.

## 4. PNG 시안 렌더링과 Vision QA 게이트

프레젠테이션 스킬의 렌더링 기능으로 모든 슬라이드를 16:9, 1280×720 PNG로 렌더링하라.
첫 렌더링은 사용자에게 공개하지 말고, Vision 기능으로 각 슬라이드를 1280×720 전체 크기에서 직접 확인하라.

다음 다섯 항목을 각각 통과 또는 수정 필요로 판정하라.

1. 메시지·컴포넌트: 제목 바로 아래에 핵심 결론·판단이 먼저 나오고 `보고 목적` 라벨이 없는지, 적용 조치마다 확인된 개선·기대효과·검증 계획 중 하나가 연결됐는지, 원문 예시가 문장·`E08`·백데이터 중 적절한 위치에 포함됐는지 확인하라. 승인된 레시피 밖의 객체가 섞이지 않았는지와 관련 하위 항목이 `G01` 같은 올바른 상위 그룹에 포함됐는지도 확인하라.
2. 줄바꿈·표기: 제목·탭·날짜·전문용어·결론이 의도치 않게 분리되지 않았는지, 날짜가 `'26.05月`·`2026-05-21`·`03/26` 중 정보 정밀도에 맞는 형식으로 반복 구간 전체에서 통일됐는지, 텍스트 잘림과 `TO-BE` 줄바꿈, 미해결 자리표시자가 없는지 확인하라.
3. 겹침: 텍스트·도형·탭·그림자·구분선·타임라인 노드가 서로 겹치거나 슬라이드 밖으로 넘지 않는지 확인하라.
4. 여백: 좌우 바깥 여백, 열 간격, 패널 내부 패딩, 제목·본문·수치의 정렬과 빈 공간의 균형이 자연스러운지 확인하라.
5. 강조: 빨간 밑줄이 강조 구문 바로 아래 2~4px에 있고 구문 폭과 비슷한지, 지나치게 길거나 짧거나 다른 내용을 가리키지 않는지, 네이비·빨강·초록·회색이 정해진 의미로만 쓰였는지 확인하라.

하나라도 `수정 필요`이면 슬라이드 번호·컴포넌트·원인을 식별하고 같은 편집 가능한 소스를 수정한 뒤 다시 렌더링하여 다섯 항목을 모두 재검수하라. 같은 문제가 반복되면 글자를 축소하는 미세 조정만 계속하지 말고 문구를 줄이거나 해당 컴포넌트를 재배치하라.
Vision QA 중 더 나은 스타일이 보여도 승인된 최상위 구조나 레시피를 임의로 바꾸지 마라. 구조 변경이 필요하면 구조 승인 단계로 돌아가라.
Vision QA는 글꼴 등록·조회와 객체 경계 검사를 대체하지 않으므로 둘을 별도로 통과시켜라.
모든 항목을 통과한 PNG 시안만 한 번에 제시하고 승인 또는 수정 요청을 받아라.
이 단계에서는 PPTX 링크나 최종 PPTX를 제공하지 마라.

## 5. 사용자 승인과 반복 수정

“승인”, “확정”, “이대로 PPTX 만들어줘”처럼 분명한 표현만 PNG 승인으로 간주하라.
질문, 모호한 반응, 일부 수정 의견은 승인으로 간주하지 마라.
수정 요청을 받으면 새 소스를 만들지 말고 기존 편집 가능한 소스를 수정하라.
수정 후 PNG를 다시 렌더링하고 검수한 다음 재승인을 요청하라.
승인 뒤 내용이나 디자인이 바뀌면 PNG 승인 상태를 해제하고 다시 렌더링하라.

## 6. PPTX 내보내기와 최종 확인

승인된 PNG를 만든 동일한 소스에서 최종 PPTX를 내보내라.
텍스트, 도형, 표, 차트가 편집 가능한지 확인하라.
최종 PPTX를 다시 1280×720 PNG로 렌더링해 승인본과 구도, 줄바꿈, 정렬이 크게 다르지 않은지 비교하고 4단계의 Vision QA 다섯 항목을 다시 통과시켜라.
차이가 크면 PPTX 또는 원본 소스를 고치고 다시 확인하라.
최종 PPTX가 글꼴을 내장하지 않았다면 수신 환경에도 나눔바른고딕 설치가 필요하다고 알려라.
검증이 끝난 뒤에만 최종 PPTX와 짧은 완료 요약을 제공하라.

## 예외 처리

- 사용자가 승인 전에 PPTX를 요구하면 PNG 승인 후 제공한다는 절차를 설명하고 현재 게이트를 진행하라.
- 사용자 템플릿이 손상됐거나 편집할 수 없으면 문제와 가능한 대안을 보고하고 선택을 받아라.
- 필요한 근거가 없으면 확인 가능한 사실과 가정을 구분하고, 근거 부족을 숨기지 마라.
- 외부 패키지가 정말 필요하면 설치 전에 필요성, 대안, 영향 범위를 설명하고 허가를 받아라.
